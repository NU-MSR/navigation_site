Task G (Simulation)

  1. We will be writing our own simulator for the turtlebot and SLAM
  2. The simulator will be in a package named nurtlesim
  3. The simulator will simulate the robot kinematics and a sensor that detects the relative x, y positions of landmarks and a landmark id

Task G.000 (Setup)

  1. Create the nurtlesim package
  2. Copy the fake_odometry.cpp node from rigid2d into nurtlesim and name it to tube_world.cpp
    • This file will be the starting point for creating the simulation environment
  3. Copy the fake_turtle_odom.launch file from rigid2d into nurtlesim and name it to tube_world.launch
    • This file will be the starting point for launching the simulation world
  4. Make sure everything compiles and runs as expected

Task G.001 (Robot Position)

  1. In addition to tracking/publishing wheel angles, the tube_world node should track the ground truth location of the robot.
  2. The following steps should occur whenever you update the wheel positions (to simulate errors in control and odometry)
    1. Add some Gaussian noise to the commanded twist (both \(v_x\) and \(\dot{\theta}\)).
    2. The robot follows the noise-corrupted twist (rather than the commanded twist) to move to its actual location.
    3. The wheel angles move to updated positions, assuming no slipping. Then wheel slip noise is added
      • You can experiment with slipping models, but a simple one is \(\eta \omega\), where \(\eta\) is uniform random noise between slip_min and slip_max and \(\omega\) is the wheel speed.
      • This method simulates wheels turning more or less than the robot actually travels, and more slipping happens when driving faster
  3. All noise values should be controllable by setting parameters for the node.
    • It is often useful to be able to set noise to zero, or set the magnitude of one noise source or another differently for testing purposes. Too bad we can't adjust noise or turn it off in the real world!

Task G.002 (Simulation World)

  1. The tube_world must also maintain the positions of obstacles in the world. The obstacles are all cardboard tubes with a 0.0762m radius
  2. Read the following as parameters:
    1. The coordinates of the tubes
    2. The radius of the tube
    3. Covariance matrix for relative x-y sensor
    4. Maximum distance beyond which tubes are not visible
  3. When the node starts, it should publish, (using a latched publisher) a MarkerArray with cylindrical markers corresponding to the locations of the tubes in the environment. This message provides the "ground truth" and allows rviz to display the location of the markers
    • Set the namespace (ns) field of the markers to real
  4. The node consistently publishes a transform between the world frame and the turtle frame to indicate the location of the robot
    • If you want, you can also publish a markers or do other things to visually depict the actual robot location
    • We are not displaying a RobotModel here because the RobotModel will be used to depict our estimated robot location.
  5. The node should publish a nav_msgs/Path on the /real_path topic. Rviz can use this message to show the actual robot trajectory
  6. At a frequency of 10 Hz, publish a MarkerArray on the /fake_sensor topic. It contains the measured positions of the tubes relative to the robot.
    • Set the namespace (ns) field for the markers
    • These positions should have noise added to them based on tube_var.
    • If a marker is not within max_range, the action for it should be set to DELETE To make it non-visible
    • You can subscribe to the markers to get these "measurements", and the measurements can also be easily drawn in rviz.
    • If you want to be fancy you can scale the marker so that it represents a level-set of the Gaussian distribution caused by the sensor noise

Task G.003 (Simulation Launch)

  1. Write a launchfile called start_sim.launch
    • It should start the simulation nodes
    • It should start rviz and display the simulation state (markers, sensed markers, the world and turtle frames, the path)
    • It should start the turtlebot3_teleop_key node to let you drive the robot. Driving the robot corresponds to changing the location of the turtle frame relative to the world frame
      • You should see the turtle frame move as you drive the robot

Task G.004 (Collision Detection)

  1. Implement collision detection between the simulated robot and the obstacles
    • Represent the robot as a circle with a given radius
    • If after updating the robot's position, the robot intersects a tube
      • Compute the line between the robot center and the obstacle center
      • Move the robot's center along this line so that the collision centers are tangent
  2. Check for collision each time you update the position of the robot actual
    • Even if there is a collision, the wheels should continue to spin
  3. You may assume that obstacles are spaced far enough apart that all collisions involve exactly one tube and the robot.

Task L (SLAM)

  • This task is about implementing Feature-Based Kalman Filter SLAM.
  • You will do this task in a new package called nuslam
  • Task L.001 - outlines the main results for the task.
  • Task L.002 - This is a subset of L.001 (implementing SLAM in a controlled environment). Once you implement this, you have completed the core algorithm.
  • I am not mandating specific nodes or parameters or a structure for you to use. The design is up to you
    • However, following the pattern we've been using in this course of keeping calculations separate from the mechanics of the nodes may prove helpful

Task L.001 (Main SLAM Results)

  • This task is about the main interface for running and visualizing your algorithms.
  • You can write whatever nodes you want, as long as you can run a slam.launch file as specified below and, where appropriate view the necessary visualizations.
  • You may also need to modify some nodes to let them publish appropriate visualization topics
  1. Write a launchfile, slam.launch robot:=X that runs your SLAM algorithm and all other necessary nodes
    • if robot == localhost The robot is simulated by your simulator. Slam will use relative x, y positions and landmark ids as sensor information
    • Later you will robot == yourturtlebot.local to run the code on the turtlebot.
    • It should also start a method to control the robot
      • I find it convenient to use the turtlebot3teleop package to drive the robot during testing
  2. Visualization
    1. If running in simulation: The simulation visualization
    2. A nav_msgs/Path showing the trajectory of the robot according only to wheel odometry
    3. A nav_msgs/Path showing the trajectory of the robot according to the SLAM algorithm
    4. The locations of the landmarks, according to SLAM (i.e., the map)
    5. The robot model itself should be displayed at the pose determined by the SLAM algorithm (using frames published by robot_state_publisher).
    6. You may find it helpful to add additional visualizations as you test and debug.

Task L.002 (SLAM in controlled environment)

  • In this task, you will test your extended Kalman Filter SLAM algorithm in the simulation environment you developed
  • This environment will enable you to verify the correctness of the algorithm without worrying about data-association or noise
  • You should be able to run the code for this task by specifying robot:=localhost with your launchfile.
  • In the next assignment, you will be using machine learning to find landmarks from laser-scanner data.
    • The data from the simulator, however, has identifiers specifying which landmark is which whereas the laser-scanner data will not have this information
  • The algorithm should compute the pose of the robots and the pose of the landmarks relative to a map frame.
  • You should publish a transform from map to odom such that map to base_footprint reflects the pose of the robot in the map frame.
    • See ROS REP 105
  1. Implement extended Kalman Filter Slam, assuming a known data association. (That is you know which measurement goes with which landmark)
  2. Drive the robot along closed path in a tube_world with several landmarks and take a screenshot of rviz at the end of your run.
    • The screenshot should clearly show the actual robot position, the estimated position, the map frame, and the odom frame
    • You should have noise enabled and a limited detection radius for this test (the radius should be small enough such that you do not see all the landmarks all the time)
    • The robot should follow a path such that it sees all the landmarks on its journey
  3. If you set to the simulator to generate no sensor noise and have a large detection radius, the SLAM path and actual path should line-up almost exactly and the landmark positions should be correct, even if the odometry noise is still present.
    • A good way to test your algorithm (in simulation!) is to drive your robot into a landmark (to make the odometry be very wrong) and watch the map estimate remain good.

Suggested Structure

The below ideas are my suggestions for how to accomplish this task; however, you may choose to ignore them.

SLAM

  1. Start with the odometer.cpp node from a previous homework. Copy it to the nuslam package and call it slam.cpp
    • In this guide, we will implement all SLAM functionality in a single node.
    • This node will also implement the odometry functionality
    • This decision makes it easier to keep everything in sync.
      • You need to be careful about synchronization. Since tf must be a tree we can't broadcast the map to base_footprint transform directly: instead, we must publish a map to odom transform such that the map to base_footprint transform is the desired value. Having all these quantities in the same node makes this process easier
      • However, you lose flexibility: odometry and SLAM are now coupled: for this project the trade-off between flexibility and complication seems worth it.
  2. Prior to implementing the algorithm, make sure the visualizations are working
    • To start out, the SLAM algorithm can pretend the odometry value is also its position estimate.
    • The inverted measurement model can provide a crude map estimate at each timestep.
    • By taking these shortcuts you have data to visualize prior to implementing the algorithm
  3. It may be helpful to create several simulated worlds with different geometry and landmarks.
  4. As an intermediate step, you can first try to implement a linear Kalman filter for 1 Dimensional SLAM and then expand upon that.

Debugging Tips

  • If your node crashes, run it in gdb as follows: rosrun --prefix 'gdb --args ' packagename nodetype [...]
  • You can also run a node in gdb from a launchfile using the launch-prefix attribute of the node tag, although I find this less useful since it is often quicker to debug a single node by recompiling and running that individual node rather than restarting the whole system
  • Once in gdb use the command r to run the program. When it crashes, use bt to see the stack trace or use up and down to navigate stack frames.
    • If a crash is in armadillo, for example, you can repeatedly run up until you get to the point in your code that called the function in armadillo that caused the crash.

Task F.999 (Turtle vs Armadillo)

  1. Download this tarball armadillo.tar.gz

  2. Copy it to the turtlebot: scp armadillo.tar.gz ubuntu@turtlebot:/tmp

  3. ssh into the turtlebot and run the following commands

    cd /tmp
tar xf armadillo.tar.gz
cd armadillo
sudo dpkg -i *.deb

  4. This installs the armadillo library, a matrix library for C++ on the turtlebot.

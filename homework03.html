Overview

  • These tasks are designed to get you a robot model that works in gazebo
  • When the gazebo model works, you should be able to easily run your waypoint following code using a gazebo simulation rather than a real robot, just by specifying robot:=-1 to your launchfiles
  • Essentially, we are laying down the infrastructure required to implement SLAM

Task A (Urdf)

Task A.004 (Inertia and Collision)

  • To work in gazebo, your urdf model needs a few changes; mainly inertia and collision geometry
  • Update the thickness of the body link of the robot to be 0.140 m (This is the distance from the bottom of the lower deck to the top of the upper deck. Having this height correct will enable us to eventually place the lidar on top of the robot)
  • The wheel axle should be 2cm from the bottom of the chassis. Make this a parameter wheel_axle_offset and update the urdf
  • Add inertia to the body and wheel of your urdf model of the robot
    • You do not need to add inertia to the caster but may if you like
    • This will require adding mass parameters for the body link and wheels
    • The turtlebot weighs 1.0kg total. Let's assume that each wheel weights 0.03 kg.
  • Add collisions to your urdf model of the robot
    • Be sure to add a collision to the caster
  • Add a link called base_scan on top of the main body of the robot (where the laser scanner goes). This link will represent the laser scanner. It can be collision-free and inertia-free.

Task G (Gazebo)

  • Task G is all about writing a gazebo plugin that enables you to control the robot from ROS.
  • We need to use tools provided by ros to extend the urdf file so it can be read into gazebo and then spawn our model into gazebo
  • To control our model we need a plugin. A plugin is a shared library written in C++ that conforms to an interface specified by gazebo
  • The plugin can call gazebo api functions as well as ROS functions

Task G.001 (Gazebo Xacro)

  • Here we will create xacro files that contain gazebo specific code to add into our model of the robot.
  • When you complete this task, running roslaunch nuturtle_gazebo =diff_drive_gazebo.launch and see your robot in gazebo
  • Create an nuturtle_gazebo package to store your gazebo related code.
    • Be sure to update the package.xml with the appropriate fields.
  • Add a diff_drive.gazebo.xacro file to the urdf folder.

    • This xacro file should use xacro:include to use nuturtle_description/urdf/diff_drive.urdf.xacro file

    • It then specifies gazebo materials for each link and friction coefficients for the wheels/caster.

      • Here is an example for the left wheel

        <mu1>0.1</mu1>
<mu2>0.1</mu2>
<minDepth>0.001</minDepth>
<fdir1>1 0 0 </fdir1>
<maxVel>0.1</maxVel>
<kp>500000</kp>
<kd>10</kd>

    • The caster should have the following options

      <mu1>0.0</mu1>
<mu2>0.0</mu2>
<minDepth>0.001</minDepth>
<maxVel>1.0</maxVel>
<kp>1000000</kp>
<kd>100</kd>

    • The minDepth is important to make friction work. Otherwise the wheels will keep "bouncing" out of the surface and the robot will move slightly.

    • Hint: URDF in Gazebo Tutorial

    • If you experience the robot drifting a little bit, the following remedies are available.

      • How well they work depend on the exact geometry of your URDF.
      • Provide a small non-zero friction coefficient to the caster (mu1 and mu2)
      • Add a small <dampingFactor> term to the wheels and/or the caster. A damping factor
      • Other factors to experiment with are here ROS URDF
  • Write a launchfile called diff_drive_gazebo.launch
    • Gazebo Launchfile Tutorial
    • Provide a launchfile argument that lets you specify a world for the robot. Default to empty_world.
    • Make sure that when the robot is spawned it lies directly on the ground
    • In gazebo, use View->inertias and verify that your inertias look correct.
      • For the body you should see a pink rectangle the same size, shape, and location of the body link
      • For each wheel you should see a rectangular box minimally enclosing the wheel

Task G.002 (Gazebo Plugin)

Overview

  • Here we write a plugin to interface Gazebo and ROS. The idea is to provide the same ROS api that the pysserial node running on the real robot provides. Thus we can substitute gazebo from the real robot.
  • To write and use a plugin we need
    1. A cpp file containing the plugin code
    2. a package.xml file that tells Gazebo where to find the plugin code
    3. A directive in the .gazebo.xacro file that tells gazebo how to load the plugin and provides parameters for it.
  • A plugin xml element cave children elements. These children are passed to the plugin at startup, which allows you to configure options for a plugin directly in the xacro file that includes it.

Instructions

Write a Gazebo plugin to control the wheel velocities of the model

  • Since the turtlebot does not have gazebo, we will not compile this gazebo package for the turtlebot

    • The first time you run catkin_arm do ./arm catkin_arm -DCATKIN_BLACKLIST_PACKAGES="nuturtle_gazebo" install. From then on (unless you delete the build directory) nuturtle_gazebo will not be built for arm)

  • Write the plugin in nuturtle_gazebo/src/turtle_drive_plugin.cpp package

  • Add the plugin to your .gazebo.xacro file (Tutorial).

    • The filename will be nuturtle_gazebo_plugin.so and plugin name is turtle_drive_plugin
    • The plugin should take xml parameters (which are provided in the xacro file for the names of the wheel joints and topics
      • <left_wheel_joint>left_joint_name</left_wheel_joint> (and for the right)
        • If the names of these joints are not specified, print a fatal error and return from load
      • <sensor_frequency>Hz</sensor_frequency> - the frequency at which to publish sensor data
        • If not specified, should default to 200 Hz
        • You can use std::stod to convert a string to a double (since c++11)
      • =<wheel_cmd_topic>name_of_wheel_cmd_topic</wheel_cmd_topic>
      • =<sensor_data_topic>name_of_sensor_topic</sensor_data_topic>

  • The plugin should subscribe to <wheel_cmd_topic> (nuturtlebot/WheelCommands)= and use that data to move the wheels.

  • The plugin should publish an <sensor_topic> (nuturtlebot/SensorData)= at the specified rate.

    • Only worry about filling in the encoder values; other values can be zero

  • The plugin should read the encoder_tics_per_rev, max_motor_rot_vel, max_motor_power, and any other parameters needed from the parameter server to know how to convert a wheel angle into encoder ticks

  • Add the following snippet of code to the Load function to make sure ROS has started

    // see http://gazebosim.org/tutorials?tut=ros_plugins Accessed 02/09/2020
// Make sure the ROS node for Gazebo has already been initialized                                                                                    
if (!ros::isInitialized())
{
  ROS_FATAL("A ROS node for Gazebo has not been initialized."
            "Unable to load plugin. Load the Gazebo system plugin"
            "'libgazebo_ros_api_plugin.so' in the gazebo_ros package");
  return;
}

Gazebo Hints

  • The Model Plugin Code here should get you started.
    • Get the example compiled and loaded into gazebo with your model prior to proceeding
    • You can add a ROS_INFO statement to the Load function. When the plugin successfully loads, you should see the message
  • More details on this process are here.

    • To avoid a warning about boost, change the cmake_minimum_requried to 3.5.4

    • Use the following code to update your CMakeLists.txt to compile the plugin

find_package(catkin REQUIRED COMPONENTS roscpp gazebo_ros [other requirements])

add_dependencies(${PROJECT_NAME}_plugin ${${PROJECT_NAME}_EXPORTED_TARGETS} ${catkin_EXPORTED_TARGETS})

    • Add the following between the <export> tags in your package.xml: =<gazebo_ros plugin_path"${prefix}/../../lib" gazebo_media_path="${prefix}" />=

  • In your plugin, in Load you want to Setup a NodeHandle and any ROS publishers and subscribers that you need
    • You are able to include ROS header files and use ROS functions as you would normally
  • Gazebo will call OnUpdate() on every simulation iteration.
    • From OnUpdate() you should publish wheel odometry and control the velocity of the wheels according to the latest received wheel_cmd command
    • The class has a Model object that can be used to manipulate the physics of the model.
    • You can see the methods available in the documentation
      • The model consists of links and joints, use GetJoint to get joints you want to control. This gives you a shared_ptr to a Joint
      • The Joint Class lets you control the joint state
      • See This tutorial to set velocities. The easiest method is to use "Joint Motors". Setting velocity directly seems to have problems when interacting with Gazebo's friction model, making it difficult to drive the car. This is because when setting velocity directly the object does not feel a force.
      • Use the getPosition function to get the position
    • Use the sdf::ElementPtr object passed to Load to get the names of the wheel joints and topics
      • The API documentation for accessing sdf elements is here
  • You don't need to spin() or spinOnce. Your plugin is being loaded into gazebo, and gazebo handles that for you.

Task S.000 (Landmark Detection)

  1. Write a node called landmarks to detect landmarks from sensor_msgs/LaserScan data
  2. The node should publish the center and radius of the landmarks it detects, using a custom nuslam/TurtleMap message on the landmarks topic

      • All the nodes necessary for running the robot and visualizing it in rviz should be started
        • turtle_interface, odometry, etc.

Hints:

  1. Writing some of the test cases in S.001 while you are coding these algorithms can help you debug and make sure you don't introduce new bugs.
    • I actually did some "test-driven-development (TDD)" and wrote the test cases for the circle detection algorithm prior to writing the algorithm itself.
  2. You can use any algorithm you want as long as it works. Below I describe algorithms that I know work.
  3. First solve an unsupervised learning problem: clustering points into groups corresponding to individual landmarks.
    • Cluster the laser scanner points based on a distance threshold.
    • Discard any clusters with fewer than 3 points, as these will not provide good results when we fit a circle
    • The laser scan data is ordered by increasing relative bearing angle. If two consecutive scan points are near each other they are part of the same cluster, otherwise they are in different clusters
    • Keep in mind, that the scan wraps around, so you may be starting in the middle of a cluster
    • To help test this step, I recommend outputting markers to help you visualize the clusters.
  4. Next, solve a supervised learning problem: circular regression.
    • To detect the circles, implement this circle fitting algorithm
    • It may help to break up each step into separate functions and test them separately.
  5. Solve a classification problem
    • Classify the clusters of points into Circle and Not Circle. This helps to avoid false positives.
    • Algorithm below is from J. Xavier et. al., Fast line, arc/circle and leg detection from laser scan data in a Player driver, ICRA 2005
      1. Let \(P_1\) and \(P_2\) be the endpoints of a cluster and \(P\) be a point in the cluster.
      2. If the cluster is an arc of a circle, then the angle from \(P_1\) to \(P\) to \(P_2\) is always the same for all \(P\) (a consequence of the inscribed angle theorem)
      3. So, compute the mean and standard deviation of the angles between the endpoints and each point on the segment
      4. If standard deviation is below 0.15 radians and mean is between 90 and 135 degrees, then it is a circle (thresholds are from the paper, tuning them will help you).
  6. Practical and Simple Extras to aid accuracy
    1. Feel free to use these ideas to increase accuracy. They will help in our specific setup but may not be as generally applicable to all situations.
    2. Eliminate circles with really large or small radii, since we know the approximate size of the landmarks (this increases the assumptions required for your algorithm to work but also is very helpful).
    3. Make the minimum cluster size larger to ensure better fits for the circle fitting algorithm

Task L.002 (SLAM in controlled environment)

  • In this task, you will test your extended Kalman Filter SLAM algorithm in a simulated environment that is completely controlled by you.
  • This environment will enable you to verify the correctness of the algorithm without worrying about data-association or noise
  • You should be able to run the code for this task by specifying robot:=-1 and debug:=true with your launchfile.
  • It should act as a stand-in for the relative landmark data returned by your landmark detection algorithm, except also carry with it information about the association of measurements to landmarks.
  • The algorithm should compute the pose of the robots and the pose of the landmarks relative to a map frame.
  • You should also publish a transform from map to odom such that map to base_link reflects the pose of the robot in the map frame.
  1. Implement extended Kalman Filter Slam, assuming a known data association. (That is you know which measurement goes with which landmark)
    • At this stage, the node only works in conjunction with the debug argument
  2. Write a node that generates the fake data from the ground truth, as reported by gazebo
    • The noise level from the fake data should be adjustable (including the ability to eliminate noise entirely)
    • There should be an adjustable radius of detection. Landmarks outside this radius are not reported.
      • You should be able to set an infinite radius
    • For later tasks, you may find it useful to be able enable/disable data-association information
    • HINT: It will be immensely helpful if the fake data and real data (i.e, the output from your landmark detection code) are compatible with each other so you can easily switch from fake data to real data without changing your SLAM algorithm.
  3. Drive the robot around a closed path in block.world and take a screenshot of =rviz at the end of the path.
    • You should have sensor noise and a limited radius for this test (the radius should be small enough such that you do not see all the landmarks all the time)
    • Report the final pose error between the actual robot position and odometry in base_git_directory/SLAM_RESULTS.md
    • Report the final pose error between the actual robot position and the SLAM estimate base_git_directory/SLAM_RESULTS.md
    • Report the landmark detection radius and noise level used.
  4. With no noise and infinite detection radius, the SLAM path and actual path should line-up almost exactly and the landmark positions should be correct.

Suggested Structure

The below ideas are my suggestions for how to accomplish this task; however, you may choose to ignore them.

Analysis

Write an analysis node to report on the ground truth from the gazebo simulation

  • The /gazebo/model_state topic publishes the actual state of models in gazebo
    • There is also a service to get this information
  • The gazebo/get_world_properties service lets you retrieve names of models.
    • All landmarks start with the world "cylinder"
  • Run analysis in the /real namespace. This way you won't accidentally rely on information you should not actually have access to in the real world
  • Messages published by analysis can be useful when debugging. I recommend creating a debug mode in your launchfiles
    • In debug mode, using topic remapping, the SLAM node will listen to the actual relative positions of the landmarks (as sent by the analysis node) rather than measurements from your landmark detection
    • Hence you can publish a real/landmarks that is a stand-in for the messages published by your landmark node

SLAM

  1. Start with the odometer.cpp node from a previous homework. Copy it to the nuslam package and call it slam
    • To ensure that the odometry is in sync with the map, this node will be used as a replacement for the odometer node.
      • You need to be very careful about synchronization. Since tf must be a tree we can't broadcast the map to base_link directly; instead, we must publish a map to odom transform such that the map to base_link transform is what we desire
    • In this guide, we will implement all SLAM functionality in a single node. While less flexible, it makes the coupling between components easier and makes it easier to see the algorithm as a single control loop.
    • One major downside is that you will be forced to run both SLAM and odometry on the same computer
    • Another downside is that this is not compatible with how ROS does navigation, which instead has a separate SLAM node that subscribes to odometry. will not be quite compatible with the way ROS typically does navigation.
      • However, I think it is easier to implement the algorithm in a single node and then separate out the SLAM node after everything is working
        • Once you get it working you could theoretically remove the odometry-related code from your SLAM node and instead subscribe to the odom message
      • If you organize everything into classes/functions it makes it easier to split up the node later if you want
  2. Prior to implementing the algorithm, get the pipeline setup and visualization set up.
    • To start out, the SLAM algorithm can use the odometry value as its estimates and the inverted measurement model as its landmark estimate.
  3. It may be helpful to create additional worlds with simpler geometry. If you update nuturtlebot I've used several worlds to debug.

Random Number Generation in C++

  • If you intend to use the C++ standard library for generating random numbers, you can use the following singleton pattern to ensure that you are only seeding the random number generator once in your process.

    #include<random>
 std::mt19937 & get_random()
 {
     // static variables inside a function are created once and persist for the remainder of the program
     static std::random_device rd{}; 
     static std::mt19937 mt{rd()};
     // we return a reference to the pseudo-random number genrator object. This is always the
     // same object every time get_random is called
     return mt;
 }

 // To generate a gaussian variable:
 std::normal_distribution<> d(mean, variance);
 d(get_random());

Debugging Tips

  • If your node crashes, run it in gdb as follows: rosrun --prefix 'gdb --args ' packagename nodetype [...]
  • You can also run a node in gdb from a launchfile using the launch-prefix attribute of the node tag, although I find this less useful since it is often quicker to debug a single node by recompiling and running that individual node rather than restarting the whole system
  • Once in gdb use the command r to run the program. When it crashes, use bt to see the stack trace or use up and down to navigate stack frames.
    • If a crash is in eigen, for example, you can repeatedly run up until you get to the point in your code that called the function in eigen that caused the crash.

Task L.003 (SLAM with unknown data association)

Here, you will implement SLAM based on the laser-scanner measurements, using the landmark detection algorithms developed in Task S.

  1. Implement a data association scheme (either based on mahalanobis distance, euclidean distance, or anything else you can think of) to associate measurements with landmarks and incorporate it into the SLAM algorithm
    • You should feel free to add any of the additional filtering techniques to make the data association more effective
    • You may wish to test your data assocation with debug:=true with the simulated relative landmark data prior to connecting it to the landmark detection algorithm.
  2. Drive the robot around a closed path and include a screenshot of the result in rviz. Use your data association scheme and simulated laser scan data.
    • Report the final pose error between the actual robot position and odometry in base_git_directory/SLAM_RESULTS.md
    • Report the final pose error between the actual robot position and the SLAM estimate base_git_directory/SLAM_RESULTS.md
  3. It is okay if your algorithm only works while driving cautiously
  4. When you complete this task you have fully implemented EKF SLAM with unknown data association
    • It may be helpful to temporarily reduce the noise of the laser scanner in the gazebo model to zero (you can do this in the xacro urdf) while testing.
  5. You may arrange the gazebo world to make this task easier for you if you wish, although it can be accomplished in the world that is provided.

Task L.004 (SLAM in the Real World)

  1. I have provided some cylindrical obstacles and walls
  2. Run your code on the turtlebot and drive it around in the real world
  3. Include a screenshot form RVIZ and a video of the turtlebot moving.
    • Report the final pose error between the actual robot position and odometry in base_git_directory/SLAM_RESULTS.md
    • Report the final pose error between the actual robot position and the SLAM estimate base_git_directory/SLAM_RESULTS.md
    • You should physically measure the actual robot position

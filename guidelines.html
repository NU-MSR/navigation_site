Homework Guidelines

Overview

These guidelines apply to all homework assignments in ME495 Sensing, Navigation, and Machine Learing for Robotics. It is also serves central location for collecting coding standards for the course. This document will be updated regularly throughout the quarter.

Important Notice

  1. Failure to adhere to certain guidelines may result in receiving no credit, even for a working solution.
  2. These guidelines apply to every homework assignment but may be overridden by instructions in an individual assignment.
  3. If you are unsure about how these guidelines apply or any assignment instructions please ask.

Homework Layout

  1. Each assignment consists of a list of tasks.
  2. Each task is given an identifier: X.YYY, where X is the group id. All tasks with the same group id are related and it makes sense to read over all the tasks in a group prior to attempting to complete any of them.
  3. Tasks within a group are not necessarily meant to be done in the order they are presented.
    • It will often make debugging easier to implement simpler versions of several of the tasks simultaneously and slowly build up functionality
    • You are encouraged to look over all the tasks before starting
  4. Some assignments are cumulative. Thus, there may be tasks with the same group id that are related/interdependent that span multiple homework assignments

Submission Instructions

  1. All homework will be completed in a single git repository
    • I will send everyone an individual link for creating this repository
  2. I will grade whatever commit is at the HEAD of default branch when I clone your repository.
  3. You should do your work on a separate branch and merge code into the primary branch when you are read for me to see it.
  4. You may push to the primary branch as many times as you would like.
    • It is possible to complete part of an assignment, push to ensure you get partial credit, then push again later when you finish

Coding Standards

The following list contains the standards that our code should strive to adhere to. Some violations will be strictly penalized, as indicated in bold.

The list may grow as the quarter progresses.

Git

  1. Improperly including files in git. For example
    • Generated executables, libraries, object files of any kind
    • URDF files generated from xacro (Automatic Zero)
    • Files generated by the catkin build system
    • The build, devel, or install space (or their *isolated counterparts) (Automatic Zero)
  2. Your name/email address not being set properly in git
  3. Each homework assignment should consist of many small git commits.
    • You should include a meaningful commit message with each commit
    • It is okay to commit code with compiler errors to your own development branch

C++

  1. Compiler error (Automatic Zero)

  2. Compiler warning

  3. Segfault

  4. No include guard in header file

  5. All .cpp files that are ROS nodes should start with the following comment block: (Automatic Zero)

    /// \file
/// \brief A brief description of what the file does
///
/// PARAMETERS:
///     parameter_name (parameter_type): description of the parameter
/// PUBLISHES:
///     topic_name (topic_type): description of topic
/// SUBSCRIBES:
///     topic_name (topic_type): description of the topic
/// SERVICES:
///     service_name (service_type): description of the service

  6. All functions/methods should be commented using doxygen style comments

    /// \brief Computes the factorial
///
/// \tparam T - An integral type
/// \param F - the factorial to compute
/// \returns F! (F Factorial)
/// <template typename T>
/// T factorial (T F);

  7. Classes should have a doxygen comment block describing what the class does. You should also provide comments for private members, where appropriate

  8. Be careful with abs: fabs() takes double and float, abs takes and returns int:

    • Since c++17 std::abs() works for floating point as well
    • Better yet: include a using std::abs and the correct version will be called.

  9. Use const everywhere you can

  10. All C++ code should be compiled with c++17, GNU extensions disabled and -Wall -Wextra

  11. Constructors should use initializer lists to initialize member variables

  12. Member variables should be initialized in the constructor initailizer list in the same order in which they are declared

  13. Put braces after control-flow expressions even if they are one line

    • (see goto fail)

  14. Do not put using statements at file scope in header files

    • These using statements change the behavior of all code that includes the header, including other headers that are below this one
    • Thus, if you do do this you can introduce include-order dependencies and mess up the functioning of other people's code when they include your header

  15. In most cases it is preferable to explicitly do a using for each function that you need rather than polluting the namespace by doing using namespace std

  16. Place variables in the scope closest to where they are needed

  17. If you use a global variable, make it static or put it in an anonymous namespace so it can't bee seen by other modules

    • There are rare cases where using an extern global variable might make sense, but not in this class, so don't use extern

  18. Usually single-argument constructors should be declared explicit

ROS

  1. Your code should pass catkin_lint -W2 warning/error/problem (Automatic Zero for Errors or Warnings here)
  2. Layout your code as git_repository/package
  3. Within a package: (note: if you have no files that fit in a given category, do not include the corresponding directory)
    • config/ Contains .yaml and .rviz files
    • launch/ Contains .launch files
    • urdf/ Contains .xacro, .urdf, and .gazebo files (but not if they are generated at runtime).
    • src/ Contains the source code for your ROS nodes and libraries
    • msg/ Contains message files
    • srv/ Contains service files
    • include/pkgname Public headers, if you are treating your package as a library
    • tests/ Unit tests and test scripts
  4. Launchfile should not result in errors due to code in one of your packages.
  5. Launchfile should not results in warnings due to code in one of your packages.
  6. Launchfile arguments should have a doc attribute to explain their use (Automatic Zero for Task)
  7. Any package.xml files should be kept up-to-date with the proper dependencies, a description, and author information

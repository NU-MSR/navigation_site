Levels of ROS testing

  1. Library level unit tests - these tests are independent of ROS and are used for testing individual functions and libraries.
    • Each test should be small and (ideally) run quickly
    • Ideally these tests are designed to run every time you compile (or at least every time immediately prior to merging your code into the master)
  2. Single Node unit tests - the goal is to validate the functionality of a single Node, including its publishers, subscribers, and servers
    • These tests should also be relatively small and self contained
    • They can be set up to run as part of the compilation process
  3. Integration test - these tests start multiple nodes and test how the whole system works together
    • These tests are made in the same way as single node unit tests, but fire up more ROS nodes
    • Generally invoked with rostest, may also have physical hardware present

Unit Testing in C++

  1. Unit tests using Google Test (gtest) can be done in ROS natively

    • This is the most common (and only officially supported) C++ unit testing framework in ROS
    • Catch2 is another option, but requires additional packages to work with ROS
      • I find it simpler to use and lighter on the dependencies

  2. To add a unit test, when that test does not depend on roscore

    • Create a test directory in your package

    • Write .cpp files in the tests directory using gtest

    • Add a <test_depend> on rosunit to the package.xml

    • Add the tests to CMakeLists.txt

      if (CATKIN_ENABLE_TESTING)
    catkin_add_gtest(${PROJECT_NAME}_your_testname tests/test.cpp)
    target_link_libraries(your_testname ${catkin_Libraries} gtest_main)
endif()

      Note that by linking with gtest_main you are using the default google test main() function, which is suitable when you are testing a library and not trying to test a node

  3. To add a unit test that does depend on roscore you will need to use rostest and make your test cases a ROS node

    • Add the the test as you did for a regular unit test

    • Implement your own main in the test code, as follows

      int main(int argc, char * argv[])
{
    testing::InitGoogleTest(&argc, argv);
    ros::init(argc, argv, "test_node_name");
    ros::NodeHandle nh;
    return RUN_ALL_TESTS();
}

    • In your CMakeLists.txt, after the initial find_package(catkin ...) add

      if (CATKIN_ENABLE_TESTING)
   find_package(catkin REQUIRED COMPONENTS rostest)
endif()

    • You then want to compile your node and have it run from a test launchfile

      if (CATKIN_ENABLE_TESTING)
    add_rostest_gtest(the_rostest tests/the_test_launchfile.test test/unit_test.cpp)
    target_link_libraries(the_rostest ${catkin_LIBRARIES} [any other libraries])
endif()

  4. To add launch or test launch files that should be run via rostest when doing a make run_tests use

    if (CATKIN_ENABLE_TESTING)
   add_rostest(tests/my_test.test)
endif()

  5. Regular launchfiles can also contain <test> nodes that are only run when invoked via rostest. This is a good way of keeping integration tests close to how the actual system is being run

  6. To actually write the tests see Google Test Primer

    #include<gtest/gtest.h>
TEST(TestSuite1, testCase1)
{
    ASSERT_EQ(2, 3);
    // There are many other ASSERT (fatal) and EXPECT (non-fatal, tests continue)
    // macros in google test
    // When first setting up tests, I like to write one that fails,
    // to ensure it is being run at the appropriate time
}

  7. To run the tests use catkin_make run_tests

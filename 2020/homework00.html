Task Group A (Robot Description)

The goal of this task group is to create descriptions of robots suitable for simulation. This task group will be spread over several assignments.

Task A.000 (nuturtle_description package)

  1. Create a ros package called nuturtle_description.
    • This should be a directory within your repository (i.e., <reponame>/nuturtle_description)
    • The package will contain urdf files and basic debugging, testing, and visualization code for the robots you will be using in this class
    • Update the package.xml as follows
      • Give it a version number of 0.0.1
      • Provide a description other than the default.
      • Fill in your name and email address as the maintainer and as an author
      • Choose a license other than TODO
  2. The package must pass catkin_lint -W2 with no warnings or problems (you will likely need to revisit point this as you add to the package).

Task A.001 (Yaml File)

  1. Create diff_params.yaml file to provide a complete parametric description of a differential drive robot.
  2. You should have the following parameters:
    • wheel_width: The width of the wheels
    • wheel_radius: The radius of the wheels
    • wheel_base: The distance between the centers wheels
    • chassis_length: The length of the main chassis link
    • chassis_thickness: The thickness of the plate that forms the chassis
  3. Include comments in the .yaml file explaining what each parameter does.
  4. The parameters above completely describe the geometry of the robot.

Task A.002 (Xacro File: Visual Only)

  1. Create a diff_drive.urdf.xacro file to describe a differential drive robot.
    • The name of the robot should be diff_drive
  2. The diff_drive.urdf.xacro file should retrieve the robot geometric properties from diff_params.yaml
    • There should be no hard-coded dimensions
  3. Each link should have properly defined visual properties
    • You do not need to provide inertial or collision properties for this task
  4. The body of the robot should be called base_link and consist of a single rectangular plate.
    • The size of this plate should automatically adjust when wheel_base changes.
    • Note: The width of the plate is not exactly wheel_base since the wheel_base is defined to be the distance between wheel centers.
  5. The base_link reference frame should be defined as follows:
    • The frame origin is centered between the wheels and wheel_radius from the rear edge of the chassis
    • The frame x-axis points in the forward direction of the robot
    • The frame z-axis points up
  6. The robot should have a third "ball caster" for support.
    • Changing wheel_radius should automatically adjust the height of the caster so that the chassis lies flat when placed on level ground
    • The caster can be implemented as a sphere that is tangent to the bottom of the chassis
    • The caster should be one "caster_radius" from the front of the robot and lie on the base_link x-axis
    • The caster should be fixed to the chassis, since we do not care about how it rolls
  7. The left wheel joint should be called left_wheel_axle.
  8. The right wheel joint should be called right_wheel_axle.
  9. Each wheel's x-axis should point forward and the wheel should rotate about it's own y axis.
    • Increasing the wheel joint angle should result in a "forward" rotation of each wheel
  10. The wheel locations axis should be offset by wheel_radius from the rear of the chassis
    • Thus the back of the wheel and the back edge of the chassis are aligned
  11. The robot's main chassis link should be called base_link
  12. Use different colors for the wheels, the caster, and the chassis

Task A.003 (Launch File)

  1. Create a launchfile called view_diff_drive.launch

  2. When invoked with --ros-args it should have the following behavior

    $ roslaunch nuturtle_description view_diff_drive.launch --ros-args
  Optional Arguments:
   use_jsp_gui (default "true"): Launch the joint_state_publisher gui to publish joint angles

  3. When invoked as roslaunch nuturtle_description view_diff_drive.launch

    • rviz should launch and display your differential drive robot model. (You will need to save rviz configuration file in the config directory and load it from the launchfile).
    • There should be no errors or warnings.
    • The joint_state_publisher gui should be displayed and let you control each wheel angle.
    • There should be a Tf Display enabled. When rviz opens it should show only the base_link frame and the left_wheel frame (no other frames should be visible
    • The fixed frame should be set to base_link

  4. The launchfile should contain a comment with a brief description of what it is used for

  5. rviz should be required. So when rviz is closed all nodes launched by the launchfile should terminate

  6. The parameters defined in diff_params.yaml should be loaded into the parameter server

    • A brief description of the package
    • The roslaunch command to visualize the robot in rviz
    • A listing of every file, with a brief description of what its purpose

Task Group B (Turtle Sim)

The goal of this task is to make a turtle in the turtle simulator move in a rectangular trajectory.

Task B.000 (tsim package)

  1. Create a ros package called tsim.
    • This should be a directory within your repository (i.e., <reponame>/tsim)
    • Update the package.xml as follows
      • Give it a version number of 0.0.1
      • Provide a description other than the default.
      • Fill in your name and email address as the maintainer and as an author
      • Choose a license other than TODO
      • The package has an exec_depend on the turtlesim package
      • The package has a depend on roscpp, geometry_msgs, and std_srvs
  2. The package must pass catkin_lint -W2 with no warnings or problems (you will likely need to revisit point this as you add to the package).

Task B.001 (turtle_rect node)

  1. Write a node called turtle_rect that reads the following parameters and logs them as ROS_INFO messages
    • x - The x coordinate of the lower left corner of a rectangle
    • y - The y coordinate of the lower left corner of a rectangle
    • width - The width of the rectangle
    • height - The height of the rectangle
    • trans_vel - The maximum translational velocity of the robot
    • rot_vel - The maximum rotational velocity of the robot
    • frequency - The frequency of the control loop
    • Be sure to document these parameters in the comments in the .yaml file.
  2. When turtle_rect starts, lift the pen, teleport the turtle to the rectangle's lower left corner, and put the pen down.
    • The node should wait for the appropriate turtlesim service to be available prior to calling the necessary services
  3. After the robot has moved to the lower left corner of the rectangle, publish velocity commands at frequency Hz to cmd_vel.
    • The turtle_node from the turtlesim package should use this topic to receive its velocity commands
    • Use a feedforward control strategy: that is, assume that the turtle follows the commands exactly and compute the control inputs that should make the turtle follow the trajectory
    • You should NOT use any information received via a subscriber in your control law
  4. Hint: You may wish to use a state machine to transition between each leg of the trajectory

Task B.002 (subscriber and service)

  1. Add a service to turtle_rect called traj_reset that moves the turtle to the lower-left corner of the rectangle and restarts the trajectory.
    • You can use a std_srvs/Empty for the service type
  2. Create a custom message called PoseError, containing the following fields
    • x_error - absolute value of the x error
    • y_error - absolute value of the x error
    • theta_error - absolute value of the angular error
    • Include comments in the .msg file describing the message and each field
  3. Subscribe to the turtlesim/Pose message published by the turtle simulator. Use this subscription to get the actual position of the turtle
  4. In your main control loop, compute the error between the position the turtle would be in, had it followed your velocity commands precisely and the latest Pose value received from the turtle simulator.
    • Publish this information as a PoseError message on the pose_error topic
  5. Upon starting the trajectory (either the first time or after a traj_reset), after calling the turtle1/teleport_absolute, wait for the turtle's position to actually be correct before beginning the trajectory
    • If you used a state machine in task B.001, then you can implement the reset/startup as another state in your state machine

Task B.003 (trect.launch)

  1. Write a launchfile called trect.launch that runs the turtle simulator and sets the parameters to have the following values: x = 3, y = 2, width = 4, height = 5, trans_vel = 2, rot_vel = 1, frequency = 100
  2. Make sure the output of turtle_rect is directed to the screen so you can see the messages printed by the subscriber you implement in B.003
  3. Add a launchfile argument called "plot_gui" that defaults to true. When true, the launchfile should
    • Launch rqt_plot~to plot the values of the ~pose_error topic
    • Be sure to document this launchfile argument using the "doc" attribute

Task B.004 (documentation)

  1. Write a README.md for your package that includes the following sections

Task C (C++ and 2D Transforms)

Here you will begin to write a library called rigid2d for performing 2D rigid body transformations.

Task C.000 (library implementation)

  1. Create a new directory called rigid2d in your base git directory.
    • For this assignment, you will compile the code as a single executable. Later we will turn it into a library
  2. Download rigid2d.hpp into rigid2d. This is a header file for the library you will implement.
    • Each function you must implement is specified in the header file
    • You should implement these functions in $rigid2d.cpp
    • You may add private members to any class but not any public members
    • Note that the doxygen style comments in the header files do not need to be repeated in the implementation files
      • It is a matter of style whether to put these detailed comments in the .cpp or .hpp file

Resources

  1. For more information about the operator overloading going on in this example see (atypically good Stack Overflow post)
  2. For more information about transforms, see Lynch & Park, "Modern Robotics", Chapter 3

Task C.001 (main implementation)

  1. Write a file called main.cpp.
    • This program should prompt the user to enter two transforms: \(T_{ab}\) and \(T_{bc}\)
    • It should then compute and output \(T_{ab}\), \(T_{ba}\), \(T_{bc}\), \(T_{cb}\), \(T_{ac}\), and \(T_{ca}\)
    • Next the program should prompt the user to enter a vector \(v\) and the frame in which it is defined (either 'a', 'b', or 'c').
    • It should then output the vector in frames 'a', 'b', and 'c'
  2. Compile your program using the following command:
    • g++ -Wall -Wextra -g -std=c++17 -o rigid2d_test main.cpp rigid2d.cpp

Task C.002 (conceptual questions)

Answer the questions below in rigid2d/C002_answers.md. Copy and paste each question into the file and place your answer below it.

  1. What is the difference between a class and a struct in C++? Hint
  2. Why is Vector2D a struct and Transform2DClass? Refer to specific C++ Core guidelines (Classes and Class Hierarchies) in your answer. (You should refer to at least 2 guidelines).
  3. Why are some of the constructors in Transform2D explicit? Refer to specific C++ Core guidelines (Constructors, Assignments, and Destructors) in your answer
  4. We need to be able to normalize Vector2D objects (i.e., find the unit vector in the direction of a given Vector2D):
    • Propose three different designs for implementing the normalize functionality
    • Discuss the pros and cons of each proposed method, in light of The C++ Core guidelines (Classes and Class Hierarchies)
    • Which of the methods would you implement and why?
  5. Implement the normalize functionality using the method you chose.
  6. Why is Transform2D::inv() declared const while Transform2D::operator*=() is not?
    • Refer to C++ Core Guidelines (Constants and Immutability) in your answer

Task C.003 (adjoints and twists)

  1. Add a class or struct called Twist2D to represent a two-dimensional twist
    • You should also define operator<< and operator>> for this class
    • We follow the convention used in Modern Robotics to place the angular component prior to the linear component
  2. Add functionality to Transform2D to convert a twist using the adjoint
    • Be sure to document everything, in a manner consistent with the rest of the file
  3. Add a prompt to main.cpp that allows a user to enter a twist in addition to a vector
    • The twist is specified in the same frame as the vector
  4. Output the twist in frames 'a', 'b', and 'c'

Task C.004 (testing)

  1. Test your program to see if it works.
    • Include a test where both \(T_ab\) and \(T_bc\) include rotation and translation and \(v\) is entered in frame \(c\)
  2. Store the input used for the test described above called test1_input.txt and include in your git repository
  3. Store the expected output from the test described above called test1_answer.txt and include in your git repository
  4. Write a script called test_compile.bash that
    • Compiles your program
    • Runs it using the input from test1_input.txt
    • Compares the output to test1_answer.txt and prints "Success!" if they match or "Failure!" if they don't.
    • The script should have executable permissions
    • Hints:
      1. Running the test and completing the comparison can be (but need not) be completed in one line of bash
      2. cmp -s - file will compare file with stdin and return success (0) if they match and failure (1) if they don't
      3. a | b sends the stdout of a into the stdin of b
      4. a < file will direct contents of file into the stdin of program a
      5. a && b will execute b only if a returns successfully
      6. a || b will execute b only if a returns failure

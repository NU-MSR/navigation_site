Task C (C++ and Rigid Transforms)

Task C.005 (ROSifying rigid2d)

  1. Turn the rigid2d library into a ROS package
    • Move the rigid2d.hpp into the include/rigid2d subdirectory and add include to your include file paths in CMakeLists.txt so you can #include"rigid2d/rigid2.hpp" in files where you need to use it
    • Move rigid2d.cpp into the src/rigid2d directory: this source file is to be a module in the library
    • Rename main.cpp to rigid2d_node.cpp into src and compile this file as a regular ROS node called node.
  2. Be sure to update the compiler flags to use c++17 and -Wall and -Werror in CMakeLists.txt
  3. You should delete the executable you compiled (rigid2d_test), if it is still present
  4. After making this change, recompile and re-run the tests you implemented in C.004
    • You will now invoke the program using rosrun rigid2d rigid2d_node
  5. For the time being, we will be using this library as a catch-all for code that is generically useful.
    • For example, for any future projects that need a definition of PI, we can use rigid2d::PI

Task C.006 (Unit Testing rigid2d)

  1. Add unit tests to the rigid2d library
    • The tests should run when using catkin_make run_tests
    • There should be at least one test per public method or function method of Transform2D
      • This includes the public members and operator<< and operator>> and operator*
      • You may find std::stringstream useful when testing operator<< and operator>> (see stringstream. Note that stringstream is a type alias for basic_stringstream<char>
    • Tests with floating point results can use Floating Point Comparison assertions.

Task C.009 (Making Vector2D a Vector)

  1. Add the following operations to Vector2D (referring to Operator Overloading Rules
    • operator+= and operator+ (perform basic vector addition)
    • operator-= and operator- (perform basic vector subtraction)
    • operator*= and operator* (perform SCALAR multiplication)
      • Be careful with this, remember scalar multiplication is valid from the left or the right
    • length(Vector2D) (computer the length of the vector)
    • distance(Vector2D, Vector2D) (compute the distance between the two vectors
    • angle(Vector2D) (compute the angle of the vector)
    • Note: some of these changes affect what functions the compiler generates for us, therefore we need to add
      • A constructor that takes two doubles (x and y)
    • A default constructor that creates a zero vector.
  2. Write unit tests for each of the operations you added.

Task C.010 (Normalize angle, it may be useful)

  2. Write some static assertions to test this function

Task E.001 (DiffDrive class)

  1. Create a DiffDrive class in rigid2d. The header file is called diff_drive.hpp and implementation file is called diff_drive.cpp

    • This class will track the state of a differential drive robot as its wheel positions are updated
    • The class should be in the rigid2d namespace

  2. I've provided a partial pseudocode version of diff_drive.hpp below. You should use this as a guide to implement the required functionality (so fill in the missing pieces in the header and implement the class). Note that the code, as presented is not valid C++ code.

    class DiffDrive
{
public:
    /// \brief the default constructor creates a robot at (0,0,0), with a fixed wheel base and wheel radius
    DiffDrive()

    /// \brief create a DiffDrive model by specifying the pose, and geometry
    ///
    /// \param pose - the current position of the robot
    /// \param wheel_base - the distance between the wheel centers
    /// \param wheel_radius - the raidus of the wheels
    DiffDrive(

    /// \brief determine the wheel velocities required to make the robot
    ///        move with the desired linear and angular velocities
    /// \param twist - the desired twist in the body frame of the robot
    /// \returns - the wheel velocities to use
    /// \throws std::exception
    WheelVelocities twistToWheels(

    /// \brief determine the body twist of the robot from its wheel velocities
    /// \param vel - the velocities of the wheels, assumed to be held constant
    ///  for one time unit
    /// \returns twist in the original body frame of the
    wheelsToTwist(WheelVelocities vel

    /// \brief Update the robot's odometry based on the current encoder readings
    /// \param left - the left encoder angle (in radians)
    /// \param right - the right encoder angle (in radians)
    /// \return the velocities of each wheel, assuming that they have been
    /// constant since the last call to updateOdometry
    WheelVelocities updateOdometry

    /// \brief update the odometry of the diff drive robot, assuming that
    /// it follows the given body twist for one time  unit
    /// \param cmd - the twist command to send to the robot
    void feedforward(

    /// \brief get the current pose of the robot
    Twist2D pose()

    /// \brief get the wheel speeds, based on the last encoder update
    WheelVelocities wheelVelocities() const

    /// \brief reset the robot to the given position/orientation
    void reset(Twist2D ps);
};

Task E.003 (Waypoints)

  1. Create a module called waypoints.hpp and waypoints.cpp in the rigid2d namespace
    • Make a class called Waypoints that helps you calculate a sequence of forward and angular velocities required for a robot to travel between the waypoints
  2. The design is up to you as long as it meets the following constraints
    • The waypoints should be provided as an std::vector<Vector2D>
    • When the robot reaches the last waypoint, it should continue toward the first one
    • Rather than pre-computing a whole trajectory, there should be a method called nextWaypoint to get the next velocity.
    • Use the DiffDrive class to model the robot movement
  3. Example: Suppose I am sending 1 velocity command every 0.1 seconds.
    • Every 0.1 seconds, I should call nextWaypoint: this is the velocity that will be used to move the robot toward the waypoint
    • Waypoints adjusts its belief in the current position of the robot based on it following the given velocity command for 0.1 second
  4. Hint: it might make sense to adopt a turn and translate strategy:
    1. turn toward the next waypoint until heading is within a threshold
    2. travel to waypoint until you reach it, then target the next waypoint

Task E.004 (Odometry Node)

This tasks is to create a node that publishes Odometry messages in a standard ROS way. A guide to Odometry in ros can be found here

  1. Write a node called odometer (in the rigid2d package) that subscribes to sensor_msgs/JointState on the joint_states topic
  2. The node should have private parameters:
    • odom_frame_id: The name of the odometry tf frame
    • body_frame_id: The name of the body tf frame
    • left_wheel_joint: The name of the left wheel joint
    • right_wheel_joint: The name of the right wheel joint
  3. Every time the node receives a JointState message it should:
    1. Update its internal odometry state (perhaps using DiffDrive)
    2. Publish a nav_msgs/Odometry message
      • You do not need to fill out any covariance information (all covariance can be set to zero)
      • The pose should be specified in the odom_frame_id frame (this is the nav_msgs/Odometry/header/frame_id)
      • The velocity will be specified as the body velocity of the robot in the body_frame_id (this is nav_msgs/Odometry/child_frame_id)
    3. Broadcast the transform between odom_frame_id and the body_frame_id on /tf (use a tf2 broadcaster)

Task Group B (Turtle Sim)

Task B.005 (Turtle Waypoints)

  1. Write a new node called turtle_way in the tsim package that has the turtle follow a trajectory of user-specified waypoints
    • The waypoint x coordinates are specified in the waypoint_x parameter
    • The waypoint y coordinates are specified in the waypoint_y parameter
    • The node should also subscribe to turtle/pose and compute the error between its actual and expected trajectory
      • For this task, the trajectory must be computed without knowledge of the pose.
    • As in turtle_rect, the turtle should pick its pen up, teleport to the first waypoint, and put its pen down and
    • The robot can publish velocity commands at a fixed rate of 60Hz
    • The robot has a fixed max forward velocity of 0.5 m/s and rotational velocity 0.5 rad/s
  2. Write a launchfile called turtle_pent.launch that uses turtle_way to make a simulated turtle traverse a pentagonal trajectory
  3. You may re-use as much code from the previous assignment as you want

Task B.006 (Fake Odometry)

  1. Write a new node called fake_odom
    • The node subscribes to geometry_msgs/Twist messages on cmd_vel. Assume that, once a cmd_vel is received, the turtle follows that velocity until the next cmd_vel command
    • When a cmd_vel is received, publish fake joint state data on joint_states for left_wheel and right_wheel assuming that the turtle had been moving at the previously received cmd_vel for the whole time period.

  1. Write a launchfile called turtle_odom.launch that runs

    • fake_odom should run, reading cmd_vel being sent to the turtle from turtle_way and publishing fake joint_states

    • You will need to set parameters to rigid2d/odometer appropriately to make this work properly

Overview

  1. The ultimate goal of this assignment is to get the turtlebot to follow waypoints based on its wheel odometry
  2. We will also be comparing positioning based on
    1. Integrating encoder data (i.e., odometry)
    2. Integrating the velocity commands we send to the wheels (i.e., our forward model, fake_odometry)
    3. The actual position of the robot in the real world.
  3. Answer questions in a file called Homework2_Answers.md in the base directory of your repository
  4. Remember to refer to the guidelines when completing this homework assignment
    • The method of turning in your homework has changed slightly. Instead of tagging your submission, I will assume that whatever is at the HEAD of the master branch is your submission.
    • Therefore, you should likely work in a separate branch and only merge into master with submission-ready code (particularly, code that compiles. If I run catkin_make and a package has a compile error, I will stop grading your homework and you will receive no credit. There is no excuse for having code that does not compile in the master branch).
    • It is a good plan to clone your repository into a clean workspace and compile to make sure you have all necessary files committed to git

Guidelines

  1. You will be sharing the robots with others in the class. You should make sure that nobody else is connected when you are trying to use the robot, otherwise your code made interfere with each other.

  3. You can run the robot off a power cord, with its wheels raised above the ground when testing many of the assignments.

Task F.002 (nuturtle_robot package)

  • The goal of this task is to get up and running on the turtlebot, and to see how to do the process manually.
  • The code that is hardware-specific for the turtlebot will be kept in the nuturtle_robot package
  • First, we will setup a launchfile that makes it easy to launch nodes on the turtlebot from our computer and start the low-level processes required to make the turtlebot run.
  1. Create a new ROS package called nuturtle_robot.
  2. Create a launchfile called basic_remote.launch
    • This launchfile should have a <machine> tag describing a turtlebot.
    • The machine tag lets you run nodes on one computer from another computer
    • The name of the machine is turtlebot
    • The launchfile takes an argument called robot
      • If robot is 0, then it is assumed that the launchfile is being run directly on a turtlebot and the address of the remote machine should be localhost
      • Otherwise, the remote machine address is turtlebotX (where X == robot).
        • The user should be student to log in as student. Do not use a password, instead your ssh keys should be setup properly (see F.000)
        • The machine should use /home/student/install/env.sh as the environment loader script.
          • Essentially this script is what roslaunch uses to "source" the setup.bash on the remote computer
  3. The launchfile should also load serial_node.py form the rosserial_python package on the turtlebot
    • Set port to /dev/ACM0 and a baud of 115200.
    • This node communications with the OpenCR board to give you low-level control through ROS
    • The firmware on that board is custom for this class, so much of the turtlebot3 software manual does not apply to us (however, it is the same hardware).
    • The node entry in the launchfile should include the proper attribute to launch it on the turtlebot machine
      • Thus, if robot is 0, the node will run on localhost and if it is another number it will run on turtlebotX

Task F.003 (Working directly via ssh)

  • It is possible to build, run and test your code directly on the turtlebot. It is a good exercise to go through once, but it will hopefully motivate you to want to not rely on the slow turtlebot processor to compile your code.
  1. In your workspace, delete the build, devel, and install directories (if they exist)
    • This step is only needed for F.003, not for general use.
  2. Copy your source space to the turtlebot:
    • cd <your_workspace_dir>
    • rsync -av --delete . student@turtlebotX:/home/student/ws
      • Note, the presence/absence of a trailing slash after directory names is significant to rsync, so use the above command precisely
      • The command makes turtlebot:/home/student/install match yourcomputer:/path/to/workspace/
      • The a flag copies all directories and permissions
      • The v flag increases verbosity (you can get more or less by adding/removing more v's)
      • --delete deletes files in the destination that are not present in the source
      • rsync is great for syncing directories between machines because it will only copy files that have changed.
  3. ssh student@turtlebotX and cd into ws
  4. On the turtlebot build the workspace and run your launchfile
    • turtlebot$ catkin_make -j1 to compile your workspace
      • This may take a few minutes (it takes me about 3).
      • The -j1 option causes make to only use 1 thread. In my experience, letting it use the full number of threads can cause the raspberry pi to run out of RAM, slowing compilation to a crawl.
    • turtlebot$ source devel/setup.bash
    • turtlebot$ roslaunch nuturtle_robot basic_remote.launch robot:=0
  5. Play a sound on the turtlebot: turtlebot$ rostopic pub /sound turtlebot3_msgs/Sound "value: 0"
    • it will only play one sound even though it keeps publishing
  6. The next steps occur on your comuter
    1. Make sure you have ros-melodic-turtlebot3 installed (via apt).
      • If you needed to install it re-source the /opt/ros/melodic/setup.bash file before continuing.
      • We are mainly using this package for the definition of turtlebot3_msgs/Sound.
    2. yourcomputer$ export ROS_MASTER_URI=http://turtlebotX:11311 so that your computer will connect to the rosmaster running on the turtlebot
    3. rostopic pub /sound turtlebot3_msgs/Sound "value: 0". You should hear the sound. Keep the publisher open
    4. Use rqt_graph to save an image nuturtle_robot/images/F003.svg image of the graph (There is a button to save as svg in the upper right corner)
      • Make sure Dead sinks and leaf topics are not hidden. You should see a /serial node subscribing to /sound and /wheel_cmd and publishing /diagnostics and /sensor_data
      • You should have Nodes/Topics (all) displayed
  7. In typical operation we will be running rosmaster on our local machine (which makes setup and debugging easier), but this shows that the turtlebot can run independently

Task F.005 (Low Level Commands)

  • We will implement our own low-level control and odometry routines in ROS in a turtle_interface node
  • Typically, these routines would be immplemented on a microcontroller.
  • This node reads in a twist and converts it to specific wheel commands that make the turtlebot move
  • It also reads in raw encoder data and outputs it as joint angles and velocities
  1. Write a node called turtle_interface that
    • Subscribes to cmd_vel (geometry_msgs/Twist) and publishes wheel_cmd (nuturtlebot/WheelCommands) that will make the turtlebot3 follow the specified twist
      • Wheel commands are integer values between -256 and 256 and are proportional to the maximum rotational velocity of the motor.
      • This node should clamp its inputs so the maximum translational and rotational velocities of the robot are not exceeded
      • The node should saturate its output so the maximum rotational speed of the motor is not exceeded
      • Theoretically the max translational and rotational speed of the robot should be solely determined by the maximum motor rotational speed and the geometry of the robot. However, there is an unknown load on the wheels (due to varying friction) which means that the max speed the robot can travel is actually less than the maximum no-load speed of the motors.
    • Subscribes to sensor_data (nuturtlebot/SensorData) and publishes joint_states (sensor_msgs/JointState) to provide the angle (in radians) and velocity (in rad/sec) of the wheels, based on encoder data.

Task F.006 (ROS API Testing)

  • This task is about testing that turtle_interface actually works by using rostest to exercise its publishers and subscribers
  • Using rostest we can test the logic of this node without needing a turtlebot
  1. Write a test node called turtle_interface_test that uses rostest to test turtle_interface ROS API
    • Write a test that verifies that cmd_vel commands with no rotational component result in the appropriate wheel_cmd
    • Write a test that verifies that cmd_vel commands with no translational component result in the appropriate wheel velocities
    • Write a test that verifies that cmd_vel commands with both translational and rotational components result in the appropriate wheel velocities
    • Write a test case that verifies that encoder data on sensors is converted to joint_states properly
    • Write a rostest file called turtle_interface.test that runs turtle_interface_test and turtle_interface
      • Register these tests in CMakeLists.txt such that this node is tested when you run catkin_make run_tests
    • Hints:
      • When publishing from your test case, be sure to use a latched publisher. This ensures that the subscriber will get every message you publish, even if it subscribes after the message has been published. Otherwise, your subscriber could miss a published message
      • With a latched publisher, you should be able to publish, loop while spinningOnce, and know that eventually the subscriber will get the message, enabling you to leave the loop and test the data.
      • When writing the .test file, only use <test> node tags for nodes that are implementing the gtest framework. Other nodes should be started with the node tag. I haven't seen this documented anywhere, but as far as I can tell, running regular (non-gtest) nodes with the test tag via catkin_make run_tests causes the test system to freeze.
      • You can create a node-handle, publisher, and subscriber on a per-test basis if desired.
      • Also, you need to run catkin_make followed by catkin_make run_tests to test your code, otherwise your non-testing nodes will not be up-to-date
      • Using output="screen" in the node tag is helpful for viewing debugging output from the nodes under test

Task F.007 (Rotation in Place)

Overview

  • Here we will write a node that causes the turtlebot to rotate
  • We will compare pose estimates from our feedforward kinematic model, our wheel odometry, and physical measurements in the world, when the robot executes pure rotation

The Node

Write a node called rotation that does the following:

  1. Offer a /start service. Whenever the /start service is called it triggers the behavior below:
    • Start accepts a parameter that indicates clockwise or counter-clockwise
  2. If available, calls the /set_pose service to reset the odometry to (0,0,0)
  3. Publish cmd_vel commands that theoretically should make the robot complete 20 full rotations in the specified direction
    • After each full rotation, the robot should pause for approximately 1/20 of a full rotation time before continuing
    • Assume that the robot is moving at a velocity specified as a parameter or command-line argument (your choice).
      • We'll refer to the parameter as frac_vel, and it is the fraction of the maximum rotational velocity that the test should use. It is \(\leq 1\)
    • Choose a frequency close to but faster than 100 Hz such that (theoretically), the robot can complete a full rotation in an integer number of periods.
    • Use a roscpp timer to publish the commands at the calculated frequency
  4. After the rotations are done, publish a cmd_vel command to stop the motors
  5. Your node should read any parameters that are needed to accomplish this task form the parameter server.

The Launchfile

  1. Write a launch file called test_movement.launch.

  2. The goal is to perform experiments comparing

    1. The robot's position based on the commands it thought it sent
    2. The robot's position based on its encoder data
    3. The robot's actual position in the world (measured by you).

  3. Hint: after reading this description of the launchfile it may help to draw a diagram of the connections between the nodes.

  4. Here is what the launchfile should do:

    • The launchfile takes a robot argument and passes it to basic_remote.launch, enabling it to run nodes the turtlebot machine.
    • Including basic_remote.launch also start the low-level /serial node
      • The turtle_interface node should run on the turtlebot machine
        • It subscribes to the sensor data being published to /sensor_data
        • It publishes WheelCommands to /wheel_cmd
      • The rotation node runs on the turtlebot machine.
        • It publishes cmd_vel which the turtle_interface node subscribes to
      • The rigid2d/odometer runs on the turtlebot machine
        • It subscribes to the joint_state data being published by the turtle_interface
        • It publishes to /odom and broadcasts the tf
        • This provides the pose of the robot based on encoder data
      • The fake_diff_encoders runs on the turtlebot
        • It should subscribe to the cmd_vel and publish fake joint states to fake/joint_states
      • Run another rigid2d/odometer node on the turtlebot
        • Subscribe to fake/joint_states
        • Publish to fake/odom and a tf from fake_odom to fake_base_link
      • Hint: Using relative topic names in your nodes and namespaces in the launchfile can minimize the number of topics you need to remap
      • Use a static_transform_publisher node to publish a transform between the odom and fake_odom frames
        • The transform should be a 0 transform, thus co-locating the odom and fake_odom frames
        • we need this transform to place fake_odom and =odom in the same tf tree.
        • You can then use tf_echo to see the difference between the fake and the real odometry
      • The urdf should be loaded into the /robot_description parameter
      • A robot_state_publisher should publish the tf frames for the urdf. This should run on the turtlebot machine
      • If robot != 0,launch rviz on the local machine to display the robot relative to the fixed odom frame
        • This visualizes where the robot thinks it is based off of its odometery
        • YOu should visualize Tf frames as well as the robot model
        • It may take some time after starting up for the turtlebot to begin publishgint joint states

  5. Hint: Don't write this launchfile all at once, start with the basics, look at the rqt_graph and do some manual testing with rostopic pub.

  6. Sometimes it can be easier to see what is happening by running the launchfile from ssh directly on the turtlebot.

  7. Do you know where your ROS_MASTER_URI is? env | grep ROS will be especially helpful

Physical Testing

  1. Mark the initial position and orientation of the robot
  2. Run test_movement.launch with
    1. frac_vel = 1.0
      • Do one run with clockwise rotation, one run with counter-clockwise rotation and report the average of the results (see below)
    2. frac_vel = Something, where something is a frac_vel < 1 that someone else in class has not used
      • Do one run with clockwise rotation, one run with counter-clockwise rotation and average the results (see below)
  3. Record the following quantities in Homework2_Answers.md and add them to the spread sheet (link sent separately)
    • The value of frac_vel used (FV)
    • The actual velocity that is used (i.e. frac_vel * max_(rot|trans)_vel). (AV)
    • The expected final \((\theta, x, y)\) pose of the robot ET, EX, EY
    • The final \((\theta, x, y)\) pose of the robot according to the odometry OT, OX, OY
    • The final \((\theta, x, y)\) pose of the robot according to the fake odometry FT, FX, FY
    • The ground truth \((\theta, x, y)\) of the robot, as measured by you GT, GX, GY
    • The drift in \(\theta\), \(x\) and \(y\) per distance traveled. DT, DX, DY
      • Let error = final odom pose - ground truth pose
      • For the rotational test, drift = error / # of rotations
      • Divide by total distance traveled (for rotation this is the total number of revolutions and for translations this is the distance traveled in meters).
  4. To accomplish the above, you need to examine some message data; how you do this is up to you (logging, rostopic echo, bagfiles, custom node, etc.)
  5. You may also need to physically measure some quantities.
  6. Does using the encoders improve the odometry over estimating the pose based on the inputs? (Answer in Homework2_Answers.md)
    • If so, why (i.e., what factors are the forward model not accounting for)?
  7. Does moving slower improve the odometry? (Answer in Homework2_Answers.md)

Task F.009 (Closed-loop Waypoints)

Overview

  • The main goal of this assignment is to have your turtle follow waypoints
  • After being able to turn and rotate in a controlled manner, being able to follow waypoints is a good test that everything fits together
  • After completing this task, the turtle should move from waypoint to waypoint in the real world and you should be able to see it's pose estimate, based on wheel odometry, in =rviz
  1. Write a node called real_waypoint that
    • Reads waypoints from the parameter server (waypoint_x and waypoint_y)
    • Reads the required velocity characteristics (such as max_trans_velocity) from the parameter server
      • Also take an argument or parameter called frac_vel that is the fraction of the max speed for turning or driving.
      • You may traverse the waypoints with frac_vel < 1.0
    • Puts a persistent marker in rviz at the location of each waypoint (latched publisher may be useful here)
      • Marker Tutorial
      • HINT: The markers are drawn by publishing; however, if rviz has not started they will not be drawn, Therefore, you may want to draw the waypoint markers only upon calling the start service
      • HINT: It will be helpful if the markers are cylinders with a radius equal to the tolerance of your waypoint arrival detection code You can then test your waypoing following by visualizing in rviz
    • Has a start service that, when called, causes the robot to follow the waypoints
      • The robot should only follow one cycle and then stop.
        • HINT: Adding functionality to the waypoint class that lets you know which waypoint the robot is at and how many cycles have been completed may be useful to you.
      • The robot's odometry should be reset such that it is located at the first waypoint
    • Has a stop service that, when called, causes the robot to stop moving
    • Sends cmd_vel commands to make the robot follow the waypoints.
      • These commands should be sent at a fixed frequency
    • You may use feedback from the odometry to help improve your waypoint tracking, but ultimately the control strategy you use is up to you
    • HINT: It may be helpful to add control gains as additional parameters in your launchfile. This way you can change/tune them without recompiling
    • HINT: There may be a fair bit of tuning and experimentation with different control strategies needed to get this right.
    • HINT: It is often helpful to be able to see internal controller data/state when tuning and debugging. Consider printing debugging information or even publishing a debugging message with your control data and plotting it with rqt_plot
    • HINT: "Less is More" - it is often the case that simpler control laws can outperform more complicated ones in practice.
    • HINT: Testing the turtlebot with it's motors up on blocks is very helpful here. You will be able to see where it thinks it is based on odometry which is sufficient to verify that the logic of your controller is correct.
  2. Write a launchfile called follow_waypoints.launch that starts all the nodes necessary to make the robot follow a 5 pointed trajectory and visualize the trajectory in rviz from the perspective of the odom frame.
  3. Include a screencast of the robot following one circuit over the waypoints in rviz and a (link to a) video of the robot following the waypoints in the world. Place links to these in Homework2_Answers.md
  4. After the robot completes its cycle, how far is it from the origin?
  5. Report the total (nominal) distance traveled by the robot and the distance from the origin in the spreadsheet.

Task B (C++ and Rigid Transforms)

Task B.006 (ROSifying rigid2d)

  1. Turn the rigid2d library into a ROS package that can be compiled with catkin
  2. Use catch_ros to integrate your unit tests with ROS
    • catch_ros is released for noetic and can be installed with apt
    • Your tests should include catch_ros/catch.hpp and not define main anymore. You can then delete catch.hpp from your repository
  3. You may wish to use catkin_create_pkg to generate the package.xml and CMakeLists.txt as a starting point

Task B.008 (Making Vector2D a Vector)

  1. Add the following operations to Vector2D (referring to Operator Overloading Rules)
    • operator+= and operator+ (perform basic vector addition)
    • operator-= and operator- (perform basic vector subtraction)
    • operator*= and operator* (perform multiplication by a SCALAR )
      • Be careful with this, remember multiplication by a scalar is valid from the left or the right
    • magnitude(Vector2D) (computer the magnitude of the vector)
    • angle(Vector2D) (compute the angle of the vector)
    • Some of these changes affect what functions the compiler generates for us, therefore we need to add
      • A constructor that takes two doubles (x and y)
      • A default constructor that creates a zero vector.
  2. Unit tests are optional for these operations, but you may find them helpful
    • You can combine efforts with your classmates as long as you cite them as in Homework 1
    • It is very helpful to write-up a quick test as you are debugging and find an error

Task B.007 (Integrate a Twist)

  1. Add a function called integrateTwist to rigid2d.
    • It should compute the transformation corresponding to a rigid body following a constant twist (in its original body frame) for one time unit.
  2. Include 3 unit tests for this function
    1. One test should test pure translation
    2. One test should test pure rotation
    3. One test should test simultaneous translation and rotation

Task B.009 (Normalize angle)

  1. Add a function double normalize_angle(double rad) to rigid2d that turns any angle into the equivalent one between \(-\pi\) and \(\pi\)
  2. Unit test is optional here
    • You may combine your efforts with others in the class, as long as you cite the source as in Homework 1.
    • It is to your benefit to test this one: if you get it wrong it will be very hard to debug later.

Task Group E (Kinematics)

Task E is related to the Kinematics of wheeled mobile robots. You can add these files to the rigid2d library (but in a separate .hpp/.cpp module). You will likely not be able to complete these tasks until all of Task C is complete.

Task E.000 (Math)

  1. This task may require you to perform some mathematical calculations. Show all your derivations in a file called Kinematics.pdf in the doc folder of the rigid2d package.
    • How you generate the documentation is up to you (latex, scanned hand-written notes, etc.)
    • Refer to your notes in your code, using comments to explain the origin of the mathematical expressions you use (labeling equations with numbers helps)

Task E.001 (DiffDrive class)

  1. Create a DiffDrive class in rigid2d. The header file is called diff_drive.hpp and implementation file is called diff_drive.cpp
    • I refer to this class in other instructions for this task. You may wish to read through those to get a better sense of what this class should be doing
  2. This class models the kinematics of a differential drive robot with a given wheel base and wheel radius.
    • The purpose of the class is to handle all of the mathematical calculations for the robot kinematics without tying that in to a specific robot or control scheme
    • This separation of concerns makes testing this class easier and also allows for its re-use in different contexts
    • You do, however, have a lot of leeway in how you implement this class
  3. The class should
    1. Track the configuration \((x, y, \theta)\) of a differential-drive robot
    2. Convert a desired twist to the equivalent wheel velocities required to achieve that twist
    3. Update the configuration of the robot, given updated wheel angles (assuming constant wheel velocity in-between updates)
  4. The class should not
    • Use any ROS publishers, services, clients, subscribers or parameters
  5. Write unit tests for diff_drive, as appropriate
    • Think of simple scenarios where you know what the answer should be.

Task E.003 (Odometry Node)

This task uses the DiffDrive class to create a node that publishes Odometry messages in a standard ROS way. A guide to Odometry in ros can be found here

  1. Write a node called odometer (in the rigid2d package) that subscribes to sensor_msgs/JointState on the joint_states topic
  2. The node should read from the following parameters:
    • odom_frame_id: The name of the odometry tf frame
    • body_frame_id: The name of the body tf frame
    • left_wheel_joint: The name of the left wheel joint
    • right_wheel_joint: The name of the right wheel joint
  3. The node should also read wheel_base and wheel_radius parameters
  4. Every time the node receives a JointState message it should:
    1. Update its internal odometry state (using the DiffDrive class to keep track it)
    2. Publish a nav_msgs/Odometry message on the odom topic
      • You do not need to fill out any covariance information (all covariance can be set to zero)
      • The pose should be specified in the odom_frame_id frame (this is the nav_msgs/Odometry/header/frame_id)
      • The velocity will be specified as the body velocity of the robot in the body_frame_id (this is nav_msgs/Odometry/child_frame_id)
    3. Broadcast the transform between odom_frame_id and the body_frame_id on /tf using a tf2 broadcaster
  • The odometer node should offer a service called set_pose.
    • The service request provides the configuration of the robot.
    • The location of the odometry is reset so that the robot thinks it is at the requested configuration

Task E.004 (Fake Turtle)

In this task you will create a kinematic simulation of a differential drive robot using the DiffDrive class.

  1. The simulation is implemented in a node called fake_turtle.
  2. The node should subscribe to geometry_msgs/Twist messages on the cmd_vel topic
    • Each time a cmd_vel message is received the "wheels" of the fake_turtle start moving with a rotational velocity that achieves the desired twist.
  3. The node should publish sensor_msgs/JointState messages on the joint joint_states topic at a fixed frequency
    • These messages should reflect the current joint angles of each wheel
  4. Use the DiffDrive class to track the state of the simulated robot.
  5. The node should read from the following parameters:
    • wheel_base The distance between the wheels
    • wheel_radius: The radius of the wheels
    • left_wheel_joint: The name of the left wheel joint
    • right_wheel_joint: The name of the right wheel joint

Task E.005 (Launch Files)

  • Write a launchfile called fake_turtle_odom.launch that starts the turtlebot3_teleop_keyboard and allows you to drive the fake_turtle node with it.
  • The launchfile should use the odometer node to compute the odometry for the simulated turtle
  • The launchfile should start rviz and show the turtlebot3 model moving in the odom frame according to the odometer node.
    • You should also visualize the odometry in rviz using the Odometry display type

Task Group F (The Real Robot)

This task is about implementing some basic turtlebot functionality and getting familiar with the hardware.

Task F.000 (Turtlebot Setup)

  • Follow the Turtlebot3 Setup instructions to get the turtlebot setup.
  • You must follow these instructions even if you have previously setup your turtlebot. The configuration is different in this class.

Task F.001 (A new workspace)

  • We need an additional package to work with the lower-level hardware on the turtlebot. This tasks helps you setup your workspace with the necessary software.
  • Your workspace directory is ws.
  • The build space is ws/build.
  • The devel space is ws/devel.
  • The source space is ws/src.
  • Your repository is in ws/src/repository
  • Clone https://github.com/ME495-Navigation/nuturtlebot.git into your source space ws/src
    • This code resides in the ws/src/nuturtlebot directory.
  • Rebuild with catkin.
  • Use vcstool (install via apt as python3-vcstool) to export a .repos file.
    • vcs export > turtle.repos. Then store turtle.repos in the base directory of your git repository.
    • Other people can then download the turtle.repos and use it to duplicate the contents of your ws/src directory.

Task F.002 (nuturtle_robot package)

The goal of this task is to write a launchfile that lets you run nodes on the turtlebot from your computer.

  1. Create a package called nuturtle_robot to store the code that interacts with the turtlebot hardware.
  2. Create a launchfile called basic_remote.launch
    • This launchfile should have a <machine> tag describing a turtlebot.
    • The machine tag lets you run nodes on one computer from another computer
    • The name of the machine is turtlebot
    • The launchfile takes an argument called robot, which allows the user to specify the hostname of the turtlebot.
      • It should default to localhost meaning that the node will run on the local machine
      • The user should be ubuntu. Do not use a password, instead your ssh keys should be setup properly
      • The machine should use /home/ubuntu/loader.sh as the environment loader script.
        • Essentially this script is what roslaunch uses to "source" the setup.bash on the remote computer
        • It also sets up ROS_MASTER_URI.
  3. The launchfile should also load serial_node.py from the rosserial_python package on the turtlebot machine
    • Set port to /dev/ttyACM0 and a baud of 115200 (these are the private parameters port and baud).
    • This node communications with the OpenCR board to give you low-level control through ROS
    • The firmware on that board is custom, so much of the turtlebot3 software manual does not apply to us (however information about hardware is correct).
  4. Include a README.md in this package that briefly explains its launch files.

Task F.003 (Working directly via ssh)

  • It is possible to build, run and test your code directly on the turtlebot. It is a good exercise to go through once, but it will hopefully motivate you to want to not rely on the slow turtlebot processor to compile your code.
  1. In your workspace, delete the build, devel, and install directories (if they exist)
    • This step is only needed specifically for task F.003, but not ever again
  2. Copy your workspace to the turtlebot:
    • cd <your_workspace_dir>
    • rsync -av --delete . ubuntu@turtlebot.local:/home/ubuntu/ws
      • Note, the presence/absence of a trailing slash after directory names is significant to rsync, so use the above command precisely
      • The command makes turtlebot:/home/ubuntu/ws match yourcomputer:/path/to/workspace/
      • The a flag copies all directories and permissions
      • The v flag increases verbosity (you can get more or less by adding/removing more v's)
      • --delete deletes files in the destination that are not present in the source
      • rsync is great for syncing directories between machines because it will only copy files that have changed.
  3. ssh ubuntu@turtlebot.local and cd into ws
  4. On the turtlebot build the workspace and run your launchfile
    • turtlebot$ catkin_make -j1 to compile your workspace
      • This may take a few minutes (it takes me about 3).
      • The -j1 option causes make to only use 1 thread. In my experience, letting it use the full number of threads can cause the raspberry pi to run out of RAM, slowing compilation to a crawl.
    • turtlebot$ source devel/setup.bash
    • turtlebot$ roslaunch nuturtle_robot basic_remote.launch robot:=localhost
  5. Play a sound on the turtlebot: turtlebot$ rostopic pub /sound turtlebot3_msgs/Sound "value: 0"
    • it will only play one sound even though it keeps publishing
  6. The next steps occur on your computer
    1. Make sure you have ros-noetic-turtlebot3 installed (via apt).
      • If you needed to install it re-source the /opt/ros/noetic/setup.bash file before continuing.
      • We are mainly using this package for the definition of turtlebot3_msgs/Sound.
    2. rostopic pub /sound turtlebot3_msgs/Sound "value: 0". You should hear the sound. Keep the publisher open
    3. Use rqt_graph to save an image nuturtle_robot/images/F003.svg of the graph (There is a button to save as svg in the upper right corner)
      • Make sure Dead sinks and leaf topics are not hidden. You should see a /serial node subscribing to /sound and /wheel_cmd and publishing /diagnostics and /sensor_data
      • You should have Nodes/Topics (all) displayed

Task F.004 (Cross-compilation)

  1. As you saw in F.003, compiling code on the turtlebot itself is rather slow.
  2. For most of our tasks, we will use a docker container that has all the necessary tools and libraries for compiling arm binaries for the turtlebot on the host system.
    • This will let us compile on our own computers, which can greatly speed up the compilation process
  3. Follow the Cross compilation instructions to setup docker and the cross-compiling system.
    • We will use ./arm catkin_arm to call catkin_make inside the container
    • Use ./arm catkin_arm install to run catkin_make install inside the container
  4. Copy the install space the robot: rsync -av --delete arm_install/ ubuntu@turtlebot:/home/ubuntu/install
    • It is crucial that install all your nodes and any other files you need on the robot so they are present in the install space.
    • You should install your launchfiles and config files in addition to your nodes, this way you can run your code directly from the turtlebot, which is a useful debugging step.
    • To install all launchfiles in launch/, for example: install(DIRECTORY launch/ DESTINATION ${CATKIN_PACKAGE_SHARE_DESTINATION}/launch
    • Pay close attention to the trailing / in the above command.
  5. Running your code, from your computer
    • From your computer: roslaunch nuturtle_robot basic_remote.launch robot:=turtlebot.local should start the serial node on the robot
    • After you complete this run rostopic list to see the topics that are available
    • run rosnode machine turtlebot.local to see the nodes running on turtlebot.local run rosnode machine <yourhostname.local> to see the nodes running on your machine
    • Make sure that on your computer you have ROS_MASTER_URI set to http://<your_hostname>.local:11311 and ROS_HOSTNAME set to <your_hostname>.local
  6. Use rqt_plot to show a graph of /sensor_data/battery_voltage and /sensor_data/accelZ. Carefully pick the turtlebot up and tilt it to the side.
    • Save the figure to nuturtle_robot/images/accbat.svg

Task F.005 (Low Level Commands)

  • We will implement our own low-level control and sensor routines in ROS in a turtle_interface node.
  • You should use the DiffDrive class to perform the computations
  • This node reads in a twist and converts it to specific wheel commands that make the turtlebot move
  • It also reads in raw encoder data and outputs it as joint angles and velocities
  • The odometer node reads the joint angles and outputs the odometry
  1. Write a node called turtle_interface that
    • Subscribes to cmd_vel (geometry_msgs/Twist) and publishes wheel_cmd (nuturtlebot/WheelCommands) that will make the turtlebot3 follow the specified twist
      • Wheel commands are integer values between -256 and 256 and are proportional to the maximum rotational velocity of the motor (see Specifications).
    • Subscribes to sensor_data (nuturtlebot/SensorData) and publishes joint_states (sensor_msgs/JointState) to provide the angle (in radians) and velocity (in rad/sec) of the wheels, based on encoder data.
      • The node should saturate its output so the maximum rotational speed of the motor is not exceeded
      • Theoretically the max translational and rotational speed of the robot should be solely determined by the maximum motor rotational speed and the geometry of the robot. However, there is an unknown load on the wheels (due to varying friction) which means that the max speed the robot can travel is actually less than the maximum no-load speed of the motors.

Task F.006 (ROS API Testing)

  • This task is about testing that turtle_interface actually works by using rostest to exercise its publishers and subscribers
  • Using rostest we can test the logic of this node without needing a turtlebot
  1. Write a test node called turtle_interface_test that uses rostest to test turtle_interface ROS API
    • Write a test that verifies that cmd_vel commands with pure translation result in the appropriate wheel_cmd
    • Write a test that verifies that cmd_vel commands with pure rotation in the appropriate wheel_cmd
    • Write a test case that verifies that encoder data on sensors is converted to joint_states properly
    • Write a rostest file called turtle_interface.test that runs turtle_interface_test and turtle_interface
      • Register these tests in CMakeLists.txt such that this node is tested when you run catkin_make run_tests
    • Hints:
      • It is sometimes helpful to run the rostest file separately from catkin_make run_tests, especially when debugging.
      • By default, rostest uses a different roscore than the regular one, making debugging difficult. However, you can force rostest to use your regular rosmaster (see rostest --help).
      • When testing subscribers, use a latched publisher. This way, the subscriber will get the last message published by your test case even if it actually subscribes AFTER the message was published.
      • When writing the .test file, only use <test> node tags for nodes that are implementing the unit tests. Other nodes should be started with the node tag. I haven't seen this documented anywhere, but as far as I can tell, running regular (non-gtest) nodes with the test tag via catkin_make run_tests causes the test system to freeze.
      • You can create a node-handle, publisher, and subscriber on a per-test basis if desired.
      • Also, you need to run catkin_make followed by catkin_make run_tests to test your code
      • Using output="screen" in the node tag is helpful for viewing debugging output from the nodes under test

Write a node called follow_circle that publishes commands that let the robot drive in a circle of a specified radius at a specified speed

  1. The node should use private parameters to determine the speed and the radius of the circle
  2. The node should offer a /control service that causes the robot to travel either clockwise, counter clockwise, or stop.

Write a launchfile called odom_teleop.launch that

Task F.008 (Physical Testing)

If you run these tests in turtleland, the course is marked out for you. However, you may run these tests at home as well. The precise distances are not important.

  1. Take a video and screencast of each of the experiments and include a gif version in your README.md.

  2. The experiments also ask questions that you should answer in the README.md

  3. Drive the robot forward and backward in a straight line several times, and then stopping when the turtlebot is in its initial configuration.

    • Don't rotate the turtlebot at all, even if it means you cannot perfectly line up with the initial configuration
    • Record the final location of the turtlebot according to odometry in the README.md (in an ideal world it would be \((0,0,0)\), but don't expect perfection!).

  4. Rotate the robot clockwise and counter clockwise several times, stopping when the turtlebot is in its initial configuration.

    • Don't translate the turtlebot at all, even if it means you cannot perfectly line up with the initial configuration
    • Record the final location of the turtlebot according to odometry in the README.md.

  5. Drive the robot in a circle, clockwise and counter clockwise several times, stopping when the turtlebot is its initial configuration.

    • When you get close to stopping, you can use the teleop_keyboard to try to get the robot as close to its initial configuration as you can.
    • Record the final location of the turtlebot according to odometry in the README.md.

  6. Try one of the previous experiments again, but try to get either a significantly better or significantly worse result.

    • Was your driving able to make a difference? What did you change with your driving to accomplish this?. Answer in the README.md

ROS Considerations

  • Frequency at which laser scan data comes in and odometry comes in is different
    • typically laser scan is much slower then odometry
  • We can handle this with map and odom frames.
  • Parent relationship is map -> odom -> baselink
  • the slam node updates only the map frame
  • the odometry node updates only the odometry frame
    • So every time odometry comes in, the robot's estimate relative to odom changes, but map->odom is the same, therefore, it is automatically like "we have this map frame, now we are moving where we are in it at a small scale due to odometry
  • odometry is an uncertainty increasing move:
  • WHen the laster data comes in, slam updates the map
    • map update accounts for the odometry having changed, so immediately after map is updated, robot's position relative to the map frame is where slam says it should be
  • This keeps odom frame always smooth but drifting, but map is always corrected for this drift
